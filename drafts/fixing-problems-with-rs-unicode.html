Fixing Problems with R's Unicode

The R language has developed a rich ecosystem that extends far beyond its original focus on statistics. But there is one area with very persistent problems: Text encoding. Because Emoji are now common in just about any context (ðŸ˜±), many users have complex textual datasets and need to explicitly work with characters beyond the normal english (ASCII) set. Ignoring proper text handling in R (e.g. by deleting special characters) is no longer an option.

In this post, I'll do a deep dive into some common but wrong ways to handle unicode in R, show examples, explain where they come from and how to fix them.

Understanding unicode (1)

The first thing we need to know about unicode is that it's in fact a special library of characters (i.e. roman letters L, composite symbols as used in e.g. arabic Ø¨Û‰ or korean writing í•œ systems, picture-based characters as in chinese hanzi æ¼¢ and emoji ðŸ™‚) along with descriptions, definitions and a unique number, and also a series of different systems to store text in binary (i.e. machine-readable) form. Each of these encodings basically defines a table (or set of rules) for translating a character (such as the letter L) into a series of bits. There are three large families: UTF-8, UTF-16 and UTF-32. These differ primarily in the amount of memory that they reserve for each character. UTF-32 is the largest encoding and always uses 32 bits (or four bytes). If you know that a text file is encoded in UTF-32, it's easy to decode: Just take consecutive chunks of four bytes and look them up in the encoding table. For the umlaut letter "o", this is:

Ã¶ -> <0x000000F6> representing the four bytes 00 00 00 F6

The smaller variants UTF-16 and (the most widely used) UTF-8 are a bit more complex. In contrast to UTF-32, which is a fixed-width encoding, they can use a varying amount of bits to encode characters (making them a varying-width encoding).

Ã¶ -> 0x00F6 representing the two bytes 00 F6

A very important implication of this translation is that there is no easy way to go backwards - from bits to characters - unless you happen to known which encoding (i.e. translation table) was used. If you're looking at a series of bits, you cannot deterministically know which encoding was used to produce it.
